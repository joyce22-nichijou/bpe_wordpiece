── BPE ALGORITHM REFERENCE ───────────────────────────────────────────────────

TRAINING (Naive):
  Input:  corpus (list of sentences), target vocab_size
  Output: self.vocab (set of tokens), self.merges (ordered list of rules)

  Steps:
    1. preprocess(corpus) → word_freq
       e.g. {"l o w </w>": 3, "n e w e s t </w>": 2}
    2. Init vocab = all individual characters (including </w>)
    3. Repeat until len(vocab) == vocab_size:
       a. Count all adjacent pair frequencies across all words (weighted by word freq)
       b. Select the pair with highest frequency
       c. Merge that pair everywhere in word_freq
          e.g. ("p","p") → replace "p p" with "pp" in all word strings
       d. Add merged token to vocab
       e. Append pair to self.merges
    4. Stop when vocab_size reached

  Scoring: score(A, B) = count(AB) across entire corpus
  (just raw frequency — this is what distinguishes BPE from WordPiece)

TRAINING (Fast — O(n log n)):
  Same result as Naive, but uses:
  - heapq (max-heap via negative values) to always get best pair in O(log n)
  - Incremental update: after merging pair (A,B), only recount pairs
    that are adjacent to the newly created token AB in word_freq
  - Do NOT rescan the entire corpus every round

TOKENIZATION (inference on new text):
  Strategy: apply self.merges rules IN ORDER to the word
  (NOT greedy longest match — the merge order IS the disambiguation strategy)

  Steps:
    1. Split word into individual characters
       e.g. "lowest" → ["l", "o", "w", "e", "s", "t", "</w>"]
    2. For each rule (A, B) in self.merges:
       scan the token list and merge any adjacent (A, B) pair
    3. Return the resulting token list (strip </w> if needed)

  Why this resolves ambiguity:
    The merge order is deterministic and fixed after training.
    No matter how you "imagine" splitting a word, applying the rules
    in order always produces the same unique result.
